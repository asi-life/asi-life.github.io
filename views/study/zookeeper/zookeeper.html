# Zookeeper简单介绍

zookeeper主要用于管理，进行分布式之间客户端的管理。

znode节点，Zookeeper中数据基本单元叫做节点，节点之下包含子节点，最后以树级方式呈现。每个节点拥有唯一路径path。客户端基于Path上传节点数据，zookeeper收到后会实时通知对该路径进行监听的客户端。

场景：锁，job，配置，服务注册等。

# 客户端命令

close 关闭当前会话

connect 连接

create 创建节点

delete 删除节点

deleteall 删除节点

get 获取节点

getAcl 获取权限配置

# Zookeeper节点介绍

  1. 节点类型

a). PERSISTENT 持久化节点 使用 create 创建

b). PERSISTENT_SEQUENTIAL 持久序号节点 使用 create -s 创建，后加序号。

c). EPHEMERAL 临时节点(不可拥有子节点) 使用 create -e 创建

d). EPHEMERAL_SEQUENTIAL 临时序号节点(不可拥有子节点) 使用 create -e -s 创建，后加序号。

  1. 查询节点属性

stat /

节点属性表如下：

字段 描述
cZxid 创建节点的事务id
ctime 创建时间
mZxid 修改节点的事务id
mtime 最后修改时间
pZxid 子节点变更的事务id
cversion 表示对此znode的子节点进行的更改次数(不包括子节点)
dataVersion 数据版本，变更次数
aclVersion 权限版本，变更次数
ephemeralOwner 临时节点所属会话id
dataLength 数据长度
numChildren 子节点数(不包含子子节点)

客户端使用 -w 可以对服务端进行监听

  1. acl权限设置

ACL全程Access Control List(访问控制列表)。Zookeeper使用ACL来控制对其znode的访问。基于scheme🆔permission的方式进行权限控制。scheme表示授权模式，id模式对应值，permission即具体的增删改权限位。

a). scheme：认证模式

方案 描述
world 开放模式，全世界都可以访问
ip ip模式，限定客户端ip访问
auth 用户密码认证模式，只有在会话中添加了认证才可以访问
digest 与auth相似，区别在于auth为铭文密码，digest使用了sha-1 + base64加密后的密码

b). permission权限位

权限位 权限 描述
c CREATE 可以创建子节点
d DELETE 可以删除子节点
r READ 可以读取节点数据及显示子节点列表
w WRITE 可以设置节点数据
a ADMIN 可以设置节点访问控制列表权限

c). acl相关命令

命令 使用方法 描述
getAcl getAcl 读取ACL权限
setAcl setAcl 设置ACL权限
addauth addauth 添加认证用户

# 常用api

a). 构造参数

connectString: 连接地址+端口

sessionTimeout: 会话超时时间，该值不能超过服务端所设置的minSessionTimeout和maxSessionTimeout

watcher：会话监听器，服务端时间将会触发该监听

sessionId： 自定义会话id

sessionPasswd： 会话密码

canBeReadOnly： 连接是否为只读的

aHostProvider：服务端地址提供者，指示客户端如何选择某个服务来调用。

# zookeeper集群

主要用于读写分离和主从自动切换，集群至少三台，且最好使用奇数台机器。

配置语法：server: <节点ID>=:<数据同步端口>:<选举端口>

a). 节点ID: 服务id手动指定1至125之间的数字，并写道对应服务节点的{dataDir}/myid文件中

b). IP地址：节点的远程IP地址，可以相同，但生成环境就不能这么做了，因为在同一台机器就无法达到容错的目的。

c). 数据同步端口：主从同时数据复制端口

d). 选举端口：主从节点选举端口

  1. 集群角色说明

共有三个角色：leader(主节点)，follower(子节点)，observer(次级子节点)

角色 描述
leader 主节点，又名领导者，用于写入数据，通过选举产生，如果宕机将会选举新的主节点
follower 子节点，又名追随者。用于实现数据的读取，同时他也是主节点的备选节点，并拥有投票权
observer 次级子节点，又名观察者，用于读取数据，与follower区别在于没有投票权。

observer配置： server.3=127.0.0.1:2889:3889:observer

  1. 选举机制

a). 第一轮投票都投给自己。

b). 第二轮投给myid比自己大的相邻节点。如果超过半数，则选举结束。

选举触发：

a). 服务节点初始化启动

b). 半数以上的节点无法和Leader建立连接

服务启动状态时会在集群中寻找leader，如果找到则与leader建立连接，自身状态变成follower或者observer。当前节点状态变化为LOOKING,进入选举流程。

在集群运行期间如果有follow或observer节点宕机，只要不超过半数并不会影响整个集群服务的正常运行。但如果leader宕机，将暂停对外服务，所有follower将进入LOOKING状态，进行选举。

  1. 数据同步机制

一个是正常的客户端写入流程，另一个是新节点或者重新启动加入时的数据同步。

a). 如果客户端将消息发送给follower，该follower会信息发送给leader，leader会发送proposal请求消息给所有follower，然后follower发送ACK确认消息给leader，如果收到超过一般的follower发送了ack，leader会发送commit消息给所有follower。

b). 需要先找leader，然后把数据进行同步。通过ZXID事务ID进行比较来确认是否需要数据同步。ZXID是一个长度为64位的数字。低32位是按数字递增。任何数据的变更都会导致，低32位的数字简单加1。高32位leader周期编导，每当选举出一个新的leader时，新的leader就从本地事务日志中取出ZXID，然后解析出高32位的周期编导，进行加1，再将低32位的全部设置为0. 保证每次选举leader后，ZXID的唯一性并且时递增的。

# zookeeper 典型使用场景

  1. 分布式集群管理

a). 主动查看线上服务节点

b). 查看服务节点资源使用情况

c). 服务离线通知

d). 服务资源(CPU, 内存，硬盘)超出阈值通知

  1. 分布式注册中心

a). 服务注册：提供者上线时将自提供的服务提交给注册中心

b). 服务注销：将服务下线

c). 服务订阅：动态实时接收服务变更消息

d). 可靠：注册服务本身是集群的，数据冗余存储。避免单点故障，及数据丢失

e). 容错：当服务器提供者出现宕机，断电等情况时，注册中心能动态感知并通知客户端服务提供者的状态。

  1. 分布式JOB

保证每个时间点只有一个节点运行job，使用主从。

  1. 分布式锁

a). 创建临时序号节点(标识为读)

b). 获取所有序号比自己小的节点

c). 判断是否全部为读节点

d). 如果是则获取成

e). 否则添加子节点变更监听，进行链表监听，监听前一个节点。

# 核心源码解析

主要是 zookeeper-server

集群启动：QuorumPeerMain

启动&加载配置：QuorumPeerMain

初始化外对服务：SeverCnxnFactory，网络编程(Netty)

初始化内存数据库：ZKDatabase

初始化会话管理器：SessionTrackerImpl

初始选举：FastLeaderElection

单机启动：ZookeeperServerMain

QuorumPeer  // 检查服务状态

FileTxnSnapLog  // 构建日志管理器，检查dataDir，事务日志目录 snapDir快照目录

FileTxnLog  // 读写事务日志文件

FileSnap  // 读写快照

a). 启动流程堆栈

单机：

1. 获取config配置

2. 解析config配置，并进行赋值

3. 构造一个数据清理器，主要用于清理快照。

4. 创建服务工厂嘞，初始化对外服务，启动一个Nio，可以设置为Netty

5. 启动服务。加载数据，启动netty服务，启动admin，启动选举

b). 快照与事务日志存储结构

事务日志：每次增删改的记录都会保存在文件中

快照日志，存储了指定时间下的所有数据
